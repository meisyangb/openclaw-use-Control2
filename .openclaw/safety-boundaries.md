# 安全边界配置 (Safety Boundaries)

**目标**: 防止 AI 在自动化操作中"死亡"（系统崩溃、数据丢失、无法恢复）

---

## 🛡️ 核心安全原则

### 1. 永远保留退路
- ✅ 所有关键操作前必须创建备份
- ✅ 备份必须可验证、可恢复
- ✅ 至少保留 3 个历史备份

### 2. 渐进式变更
- ✅ 小步提交，频繁验证
- ✅ 每次只改一件事
- ✅ 变更后立即检查

### 3. 自动回滚
- ✅ 检测到错误立即回滚
- ✅ 回滚后记录原因
- ✅ 回滚后通知用户

### 4. 健康检查
- ✅ 操作前检查系统健康
- ✅ 操作后验证结果
- ✅ 定期检查系统状态

---

## 🚫 禁止操作（红线）

以下操作**必须**经过用户确认：

1. **删除操作**
   - ❌ 删除 Git 分支（特别是 main/master）
   - ❌ 删除 Git 标签
   - ❌ 删除备份文件
   - ❌ 删除配置文件

2. **强制操作**
   - ❌ `git push --force`（除非明确授权）
   - ❌ `git reset --hard`（除非回滚流程）
   - ❌ 覆盖重要文件

3. **大规模变更**
   - ❌ 单次提交超过 50 个文件
   - ❌ 单次提交超过 10MB
   - ❌ 重构核心模块（无测试）

## ✅ 安全操作流程

### 标准提交流程

```
1. 健康检查 → 失败则中止
2. 创建备份 → 失败则警告
3. 添加文件 → 验证文件列表
4. 提交变更 → 验证提交信息
5. 运行测试 → 失败则回滚
6. 更新状态 → 记录操作日志
```

### 发布流程

```
1. 健康检查 → 必须通过
2. 创建备份 → 标记为 pre_release
3. 升级版本 → 更新版本文件
4. 生成 CHANGELOG → 验证格式
5. 提交变更 → 使用安全提交
6. 创建标签 → 验证标签
7. 推送远程 → 失败则回滚
8. 创建 Release → GitHub Release
```

### 回滚流程

```
1. 检测错误 → 记录错误信息
2. 停止操作 → 设置紧急停止标志
3. 选择备份 → 使用最近的可用备份
4. 执行回滚 → 恢复文件
5. 验证恢复 → 检查完整性
6. 记录历史 → 更新回滚日志
7. 通知用户 → 报告回滚原因
8. 清理标志 → 移除紧急停止
```

|---------|---------|---------|---------|---------|
| 常规提交 | 低 | 推荐 | 自动 | 否 |

| 发布流程 | 高 | 必须 | 自动 | 推荐 |

| 核心重构 | 极高 | 必须 | 手动 | **必须** |

## 🚨 紧急停止机制

### 触发条件

1. 连续 3 次操作失败
2. 回滚失败
3. 健康检查连续失败
4. 检测到异常行为

### 停止后动作

1. 创建 `.openclaw/EMERGENCY_STOP` 文件
2. 停止所有自动化操作
3. 记录详细错误日志
4. 通知用户介入

### 恢复流程

1. 用户检查问题
2. 手动修复错误
3. 删除 `EMERGENCY_STOP` 文件
4. 运行健康检查
5. 恢复自动化

## 🔄 自我修复机制

### 自动修复场景

1. **文件权限错误** → 自动修复权限
2. **Git 锁文件** → 自动清理锁文件
3. **临时文件** → 自动清理临时文件
4. **缓存过期** → 自动刷新缓存

### 需要人工介入场景

1. **数据损坏** → 用户确认恢复
2. **配置冲突** → 用户决定策略
3. **依赖问题** → 用户更新依赖
4. **权限问题** → 用户提升权限

## 💡 最佳实践

### 日常操作

1. **频繁提交** - 小步快跑，降低风险
2. **及时备份** - 操作前必备份
3. **验证结果** - 操作后必检查
4. **记录日志** - 便于追溯问题

### 发布前检查清单

- [ ] 健康检查通过
- [ ] 所有测试通过
- [ ] CHANGELOG 已更新
- [ ] 版本号已升级
- [ ] 备份已创建
- [ ] 回滚方案就绪
- [ ] 通知已发送

### 事故后复盘

1. 记录事故经过
2. 分析根本原因
3. 更新安全策略
4. 改进自动化流程
5. 测试修复方案
